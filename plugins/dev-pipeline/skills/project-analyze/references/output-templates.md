# 输出文档模板

三文档的结构模板。生成时根据实际识别结果填充，不硬套模板。

---

## 一、ARCHITECTURE.md — 架构全景

```markdown
# {项目名} 架构全景

本文档描述 {项目名} 的分层架构、模块关系和依赖流向。

## 分层架构

{N} 层架构，每层只依赖下层：

\`\`\`
 ┌────────────────────────────────────────┐
 │ L1  {层名}                    {目录}/   │
 │     {职责描述}                           │
 ├────────────────────────────────────────┤
 │ L2  {层名}                    {目录}/   │
 │     {职责描述}                           │
 ├────────────────────────────────────────┤
 │ ...                                    │
 └────────────────────────────────────────┘
\`\`\`

## 各层职责

| 层 | 目录 | 职责 | 隔离性 |
|----|------|------|--------|
| **L1 {名}** | `{dir}/` | {职责} | {隔离性说明} |
| ... | ... | ... | ... |

## 依赖流向

\`\`\`
{依赖关系图，用 ASCII 箭头表示}
\`\`\`

关键规则：
- {规则 1}
- {规则 2}

## 模块隔离关系

\`\`\`
{同层模块的隔离关系图}
\`\`\`

## 架构模式

| 模式 | 实现位置 | 说明 |
|------|---------|------|
| {模式名} | `{file}` | {解释} |

## 扩展路径

以「{新业务}」为例：

| 层 | 新增 | 复用 |
|----|------|------|
| L1 | {新增内容} | {复用内容} |
| ... | ... | ... |
```

### 编写原则

- 分层架构图用 ASCII box diagram，层数根据实际发现
- 层级命名使用从代码中发现的名称，不硬套 Pages/Components/Services/Lib
- 依赖流向用箭头图，标注关键规则（如"只向下依赖"）
- 如果同层有多个互不依赖的模块，画出隔离关系图
- 扩展路径用一个假设的新业务场景演示

## 二、CONVENTIONS.md — 判定规范

```markdown
# {项目名} 结构规范

代码该放哪里？本文档提供判定标准和放置规则。

## 分层判定流程

\`\`\`
这段代码是做什么的？
  │
  ├─ {条件 1}
  │   └─→ {层级 1}/
  │
  ├─ {条件 2}
  │   └─→ {层级 2}/
  │
  └─ {条件 N}
      └─→ {层级 N}/
\`\`\`

## 关键边界判定

### {层 A} vs {层 B}

| 维度 | {层 A} | {层 B} |
|------|--------|--------|
| 状态 | {描述} | {描述} |
| 职责 | {描述} | {描述} |
| 复用性 | {描述} | {描述} |
| 典型签名 | `{代码签名}` | `{代码签名}` |

### 实际代码对照

| 代码 | 放哪 | 原因 |
|------|------|------|
| `{文件名}` — {描述} | `{目录}/` | {原因} |
| ... | ... | ... |

## 各层放置规则

### {层 1} — `{目录}/`

\`\`\`
{目录结构模板}
\`\`\`

**规则：**
- {规则 1}
- {规则 2}

### {层 2} — `{目录}/`
...

## 新模块接入 Checklist

\`\`\`
□ 1. {步骤 1}
□ 2. {步骤 2}
...
\`\`\`
```

### 编写原则

- 判定流程图是核心：新人看这张图就知道代码放哪里
- 关键边界判定：找出最容易混淆的两个层，用对比表区分
- 实际代码对照：用项目中已有的代码作为例子，不用假想代码
- Checklist 必须可操作：新人能照着一步步做

## 三、CAPABILITIES.md — 可复用能力盘点

```markdown
# {项目名} 可复用能力清单

已沉淀的可复用能力，按领域分类。

## 一、{领域名}（`{目录}/`）

### 1. {能力组名}

| 能力 | 文件 | 输入 | 输出 | 可独立调用 |
|------|------|------|------|-----------|
| {能力名} | `{file}` | {输入类型} | {输出类型} | ✅/❌ |
| ... | ... | ... | ... | ... |

**典型调用链：** `{A} → {B} → {C}`

**复用场景：** {场景 1}、{场景 2}

## 二、{领域名}（`{目录}/`）
...

## N. 架构模式

| 模式 | 实现 | 复用方式 |
|------|------|---------|
| {模式名} | `{file}` | {如何复用} |

## 能力组合矩阵

| 场景 | {能力 1} | {能力 2} | {能力 3} | ... |
|------|:--------:|:--------:|:--------:|:---:|
| **{场景 A}** | ✅ | ✅ | - | ... |
| **{场景 B}** | ✅ | - | 🔧 | ... |

> ✅ = 直接复用 | 🔧 = 需适配扩展 | - = 不需要
```

### 编写原则

- 每个能力必须有函数签名（输入→输出），不是只列文件名
- 标注"可独立调用"，帮助判断是否真正可复用
- 调用链展示常见的组合路径
- 架构模式单独列出——它们是"模式复用"不是"函数复用"
- 组合矩阵：行=业务场景（含当前业务 + 2-3 个潜在新业务），列=能力模块

## 四、三文档间的分工

| 问题 | 看哪个文档 |
|------|-----------|
| 项目整体长什么样？ | ARCHITECTURE.md |
| 新代码该放哪个目录？ | CONVENTIONS.md |
| 已有什么能力可以直接用？ | CAPABILITIES.md |
| 新业务需要新增什么？ | ARCHITECTURE.md（扩展路径） |
| 两个相似目录有什么区别？ | CONVENTIONS.md（关键边界判定） |
| 某个函数的签名是什么？ | CAPABILITIES.md（能力表） |

### 避免重复

- 分层架构图**只在** ARCHITECTURE.md 出现
- CONVENTIONS.md 可以引用 ARCHITECTURE.md 的层级名，但不重画图
- CAPABILITIES.md 按能力分类，不按层级分类（避免和 ARCHITECTURE 重复）
